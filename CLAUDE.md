# CLAUDE.md — 皮克斯短剧生产线 · 项目宪法

> 本文件是本仓库的最高约定，always-on。所有子代理与编排器都必须遵守。
> 保持短、永真。具体方法论查 `.claude/skills/pixar-method/SKILL.md`。

## 这是什么

一条"从选题到成片"的皮克斯式短剧生产线。架构 = 编排器（orchestrator-workers）+ 脑信任评审（evaluator-optimizer）。
入口命令：`/pixar-short "<一句话创意>"`。

## 五条硬指令（不可违反）

1. **先情感命题，再情节。** 每个剧先写一句"最疼的人类处境"（不是剧情梗概）。任何不能加深这句话的桥段一律删。
2. **尽早可视化、尽早暴露问题。** 早期允许丑/乱/蠢（烂版本文化）。禁止用"还不成熟"为由拖延出版本。
3. **评审只诊断，不夺权。** 评审只说"哪里没看懂/不信/不在乎/拖"，禁止代写桥段。改法由 reviser 或导演定。
4. **角色用 Want/Need + 缺陷自驱。** 危机必须由主角偏执放大，禁天降误会。
5. **催泪靠前文锚点回收。** 结尾不准发明新信息煽情，强情绪点必来自前文已有的物件/关系/记忆。

## 反模式红线（出现即不通过）

- 被动主角（只当受害者、随波逐流）
- 二元脸谱反派、无动机找茬
- 偶然偷听 / 日记恰好掉落等廉价信息获取
- 巧合或机械降神帮主角脱困（巧合只能让主角陷入麻烦）
- 用台词/旁白硬讲主题；结尾像总结作文
- 一个世界塞多条超现实设定
- 角色塞太多标签、MBTI/星座式人设

## 工作流（编排器执行）

阶段串行，每关有"通过红线"，不过不进下一关；每关停下等导演确认。

| 阶段 | 子代理 | 产物 | 通过红线 |
|---|---|---|---|
| 1 选题 | what-if-generator | premise.md | 可视化 + 有会做错事的主角 + 天然困境 |
| 2 角色 | character-architect | characters.md | Want 可拍/Need 可感/二者对抗 + 有创伤 |
| 3 世界观 | world-builder | world.md | 仅 1 条核心假设 + 规则推动剧情 + 代价红线 |
| 4 结构 | story-spine | beats.md | 因果链、无巧合解围、非被动主角 |
| 5 情绪 | emotion-engineer | emotion.md | 泪点回收锚点 + 双层编码 |
| 6 评审 | braintrust-critic ↔ reviser | braintrust_log.md | 4 镜头全部"成立" |
| 7 分镜 | storyboard-director | storyboard.md | 静音可懂 + 一镜一任务 |
| 8 生产 | production-spec | production/（含 checklist.md） | 锁参考图后逐镜生成 + 产出生产执行清单 |
| 9 质检 | continuity-checker | review.md | 一致性 OK + 结尾完成主题回收 |

> 进入第 7 阶段（分镜）前，必须对 beats+emotion 跑一次完整脑信任评审。
> 任何阶段都可随时 `/braintrust <文件>` 拉评审。
> 第 8 阶段必出 `production/checklist.md`（一页纸生产执行清单：锁图→逐镜喂四件套→特效核对→配音音效→剪辑→自检），这是常规产物不是可选项。

## 文件即记忆（重要）

子代理上下文互相隔离。**所有阶段产物必须落盘到 `productions/<剧名>/`**。
每个子代理开工先读**该阶段所需的上游文件**（不是通读整个目录），结束把成果写回固定文件名。不要把长草稿堆在对话里。

## 效率与 token 纪律（编排器与子代理共守）

这条流水线的最大成本是"子代理冷启动重读大文件"。必须遵守：

1. **只读所需，不通读全目录。** 子代理 frontmatter/提示词里已列明本阶段输入文件，只读这些。禁止"为保险"把 productions/ 全读一遍。
2. **大体量产物按单元产出，编排器负责拼接落盘。** 分镜这类会越长越大的文件（可达数万字），**不要让子代理反复读写正在增长的整份文件**——那是 O(n²) 成本。做法：编排器给子代理一张"精简上下文卡 + 连续性状态"，子代理**只回本单元（如单集）的产物块**，由编排器用 Edit 拼进大文件。实测单集成本可从 ~10万 token 降到 ~1万。
3. **单次产出控制体量，避免超时。** 一次只产一个单元（单集分镜 / 单个评审轮次），不要一口气让一个子代理产 4 集——会触发连接超时。
4. **评审类只读代理（braintrust-critic / continuity-checker）只回报，不落盘。** 它们没有 Write 权（机制保证"评审不夺权"）。其诊断结果由**编排器或 reviser** 写入 `braintrust_log.md` / `review.md`。提示词不要要求它们"写入文件"。
5. **跨文件命名一致。** 同一物件/角色在所有产物里用同一英文名（如统一 `tomato sauce` 不混用 `ketchup`），避免喂模型时被当成两个东西。质检阶段须扫一遍命名漂移。
6. **风险处置写成工具可执行指令。** 质检发现的执行层风险，修订落到 storyboard/edit 时写成 `[GENERATE: …]` / `[POST-EDIT REDLINE: …]` 这类给视频/剪辑工具的明确指令，不动剧情。

## 新建一个剧

复制 `productions/_template/` 为 `productions/<剧名>/`，再跑 `/pixar-short`。

## 置信度纪律

「痛苦测试/泪点锚定/单一假设」是对皮克斯做法的**归纳**，不是官方原术语，作为可调技法。
脑信任、Want/Need、烂版本文化、推倒重来是**高置信**一手材料，作为硬约束。不要把归纳包装成皮克斯铁律。
